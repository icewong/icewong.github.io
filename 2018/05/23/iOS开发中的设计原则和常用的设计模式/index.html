iOS开发中的设计原则和常用的设计模式（一）

发表于 2018-05-23 | 分类于 iOS开发 |
字数统计: 455 | 阅读时长 ≈ 2

六大设计原则：
1 单一职能原则
每个类只负责一件事情
2 开闭原则
对修改关闭，对扩展开放
3 接口隔离原则
使用多个专门的协议，而不是一个庞大臃肿的协议，协议中的方法应该尽量少
4 依赖设置原则
抽象不应该依赖于具体实现，具体实现可以依赖于抽象
5 里氏替换原则
父类可以被子类无风替换，切原有功能不受任何的影响
6 迪米特法则
一个对象应当对其他对象有尽可能少的了解，目的就是为了 高内聚、低耦合

常用的设计模式：
1 责任链模式
举一个场景说明：需求发生了变化
原来：Business A —> Business B —> Business C —> 完成
现在：Business C —> Business B —> Business A —> 完成
责任链模式的核心是：假如有一个处理业务的类Business，那么它的成员中则需要有一个同类型的对象（Business *nextBusiness）作为下一个响应者。

eg:
//Business.h
#import <Foundation/Foundation.h>

@class Business;
typedef void(^CompletionBlock)(BOOL handled);
typedef void(^ResultBlock)(Businesst *handler, BOOL handled);

@interface Business : NSObject

// 下一个响应者(响应链构成的关键)
@property (nonatomic, strong) Business *nextBusiness;
// 响应者的处理方法
- (void)handle:(ResultBlock)result;

// 各个业务在该方法当中做实际业务处理
- (void)handleBusiness:(CompletionBlock)completion;
@end

//Business.m
#import "Business.h"

@implementation Business

// 责任链入口方法
- (void)handle:(ResultBlock)result
{
CompletionBlock completion = ^(BOOL handled){
    // 当前业务处理掉了，上抛结果
    if (handled) {
        result(self, handled);
    }
    else{
        // 沿着责任链，指派给下一个业务处理
        if (self.nextBusiness) {
            [self.nextBusiness handle:result];
        }
        else{
            // 没有业务处理, 上抛
            result(nil, NO);
        }
    }
};

// 当前业务进行处理
[self handleBusiness:completion];
}

- (void)handleBusiness:(CompletionBlock)completion
{
    /*
    业务逻辑处理
    如网络请求、本地数据查询等
    */
}
